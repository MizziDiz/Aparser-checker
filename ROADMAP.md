# Роадмап

## ✅ В работе / готово частично

### Статистика в SQLite (`--stats`) — стартовала
`lib/stats.py` собирает в `data/aparser_stats.db`:
- **доменные зоны** из файлов результатов (URL по строке → суффикс через `tldextract`);
- **операторы запросов** из query-файлов (`site:.com`, `instreamset:(url):.org`, `inurl:…`).
Инкрементально: каждый результат разбирается один раз (по сигнатуре mtime:size).

Дальше по этому пункту:
- [x] снимки метрик заданий из карточек (status/done/total/failed%/speed/results) в ту же БД — time-series (`task_snapshots`), пишутся в проходе монитора для прогрессирующих заданий;
- [x] ретенция и оптимизация: удаление данных старше `stats_retention_days` (не чаще раза в 6 ч), WAL, индексы;
- [ ] интерфейс к статистике (вкладка в exe / веб-дашборд) — пока без интерфейса, только сбор;
- [x] топ-сводки в Telegram (`--top`): топ зон/операторов за `top_period_days` дней, `top_limit` позиций.

### Слежение за остатком и ETA — готово
Остаток = total − done по карточкам; скорость и ETA считаются по снимкам `task_snapshots`
(Δdone/Δвремя за окно `eta_window_min`, единицы-независимо). Уведомление «⏳ почти готово»
при `almost_done_pct`; остаток+ETA добавляются в сводку/heartbeat.

### Telegram-уведомления — обогатить под конвейер

### Режимы управления парком: локальный и централизованный
Сейчас каждый сервер автономен: свой конфиг/состояние/БД, монитор и автопилот крутятся
локально, наружу — только сообщения в Telegram (напрямую или через релей). Нужно два
чётких режима развёртывания, с общей кодовой базой.

**A. Локальный (текущий, довести до продукта).** Узел сам себя мониторит, генерит ключи
(`--keygen`), создаёт задания (`--autopilot`), шлёт результаты (autosend) и статистику.
Ноль зависимостей от других узлов, кроме релея для Telegram. Осталось: упаковать в
удобное развёртывание (exe/служба), единый конфиг, самодиагностика.

**B. Централизованный (control plane).** Выделенный сервер-контроллер управляет парком.
Готово: `--config <узел>` ведёт несколько узлов из одной точки (per-node state/лог),
таймауты UI под удалённый доступ. Дальше:
- [ ] **инвентарь узлов**: список серверов (адрес в VPN, роль, версия, статус) + heartbeat
  от каждого; хранить в БД контроллера;
- [ ] **транспорт до узлов** поверх приватной сети (VPN): единый способ дотянуться до
  web-UI/файлов/агента каждого узла;
- [ ] **агент на узле**: тонкий HTTP/же-релей-подобный сервис, принимает команды от
  контроллера (собрать статус, запустить `--keygen`/`--autopilot`/`--stats`, отдать лог/БД);
  переиспользовать модель `lib/relay.py` (секрет + allowlist IP);
- [ ] **распоряжение ключами централизованно**: генерить батчи на контроллере и
  раскладывать их на узлы (в их `queries/`), либо по запросу узла — «выдай работу»;
- [ ] **сводный дашборд/сводки**: агрегировать статистику и остаток/ETA по всем узлам
  в одном месте (Telegram-сводка по парку + веб/exe-панель поверх `config_schema`);
- [ ] **безопасность**: только приватная сеть (VPN), секрет+allowlist на агенте,
  никаких публичных портов; ротация ключей.

> Конкретные параметры развёртывания (VPN-эндпоинты, адреса узлов, роли) держатся вне
> репозитория — в приватных заметках оператора (`data/ops/`), не в этом публичном репо.
