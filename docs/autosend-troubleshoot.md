# Роадмап: autosend не шлёт файлы (серверы 13/14)

**Симптом:** мониторинг работает, `--test-telegram` проходит, но результаты не
отправляются на шару. Ошибок в Telegram нет — файлы «просто не уходят».

**Причина класса проблемы:** «тест пересылки» (`--test-telegram`) проверяет только
канал Telegram. Он НЕ проверяет реальный отбор файлов: сопоставление задача↔результат,
порог `settle`, журнал уже отправленных. А в `run_autosend` все причины пропуска
писались в `logger.debug` — при обычном уровне логов их не видно. Поэтому autosend мог
молча пропускать всё, а «тест» — проходить.

## Что уже сделано (видимость причины)

1. **Итоговая строка в каждом прогоне** (`lib/autosend.py`, INFO, всегда):
   `autosend: задач N, отправлено K, уже были …, без результата …, устаканиваются …, удалено …`
   — теперь обычный лог cron на 13/14 сразу показывает, почему отправлено 0.
2. **Режим `--autosend-check`** — сухой прогон, ничего не копирует/не удаляет. По каждой
   задаче печатает вердикт (ОТПРАВИТ / НЕТ РЕЗУЛЬТАТА / УСТАКАНИВАЕТСЯ / УЖЕ ОТПРАВЛЕН),
   плюс проверяет: заданы ли 3 пути, существуют ли queries/results, **доступна ли шара на
   запись** (проба записи временного файла). Есть в обеих точках входа:
   ```
   python aparser_monitor.py --autosend-check       # Enterprise-ветка
   python aparser_monitor_ui.py --autosend-check     # Pro-ветка
   ```

## Порядок диагностики на 13/14

1. Запустить `--autosend-check`. Итоговая строка + первая же метка укажут причину ниже.
2. Сверить с гипотезами и применить фикс.

## Гипотезы (по убыванию вероятности) и что покажет `--autosend-check`

| # | Причина | Признак в `--autosend-check` | Фикс |
|---|---------|------------------------------|------|
| H1 | В конфиге 13/14 заполнены не все 3 пути (`queries_dir`/`results_dir`/`autosend_dest`) → `run_autosend` выходит сразу | `❌ autosend ВЫКЛЮЧЕН: один из трёх путей пуст` | Заполнить недостающий путь в `data/aparser_monitor.config.json` |
| H2 | Имена файлов результатов НЕ совпадают с именами задач (иная структура results на этих серверах) → `_find_result` не находит ничего | Все задачи `⏳ НЕТ РЕЗУЛЬТАТА` + предупреждение об именах | Сверить реальные пути/имена в `results_dir`; при иной схеме — расширить сопоставление в `_find_result` (напр. по basename без учёта расширения / по шаблону) |
| H3 | `results_dir` в конфиге указывает не туда (напр. дефолт `C:\A-Parser\...`, а на 13/14 другой путь установки) | `results_dir существует: False` или все `НЕТ РЕЗУЛЬТАТА` | Поправить `results_dir` под реальную установку сервера |
| H4 | Шара недоступна/нет прав записи с 13/14 (но копирование тогда даёт OSError в лог — проверить лог) | `dest доступен на запись: False → <ошибка>` | Проверить UNC-путь, креды, что служба/юзер задачи видит шару |
| H5 | Всё моложе `settle` (или расхождение часов/mtime) → вечно «ещё пишется» | Многие `⏱ УСТАКАНИВАЕТСЯ` | Проверить `autosend_settle_min`, время на сервере; при стабильных файлах уменьшить порог |
| H6 | Журнал `aparser_sent.jsonl` уже помечает версии отправленными (перенос состояния/копия узла) | Все `✅ УЖЕ ОТПРАВЛЕН`, «отправит 0» | Если ложно — очистить/пересоздать журнал; убедиться, что `data/` не скопирован с другого узла |
| H7 | На 13/14 планировщик зовёт autosend не тем способом (напр. отдельная задача `--autosend`, а в её конфиге нет путей; или monitor-проход не доходит до autosend) | `--autosend-check` работает, а плановый прогон — нет → сверить команду в schtasks | Выровнять команду/конфиг планировщика с рабочими серверами |

## Дальнейшие шаги (после установления причины)

- [ ] Прогнать `--autosend-check` на 13/14, зафиксировать какой из H1–H7.
- [ ] H2/H3 (наиболее вероятны для «молча ничего»): сверить реальную схему `results` на
      13/14 с рабочим сервером; при необходимости — обобщить `_find_result`
      (сопоставление по basename без расширения + опциональный `result_glob` в конфиге).
- [ ] Отбить регрессию: добавить в итоговую Telegram-сводку/heartbeat строку autosend
      (`отправлено/без результата`), чтобы «0 отправок при наличии задач» было видно
      удалённо, без захода на сервер.
- [ ] Опционально: `--autosend-check --json` для централизованного сбора статуса узлов.

> Заполнять образцами с реальных серверов: точную схему `results` на 13/14 (дерево
> `results_dir` — как называются файлы результатов относительно задач) — это снимет
> неоднозначность H2/H3 окончательно.
